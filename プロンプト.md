## レビューしてほしい観点

以下の観点でレビュー・フィードバックをください。

### 1. 整合性チェック

- **product.md ↔ data-model.md ↔ backend.md** の間で矛盾している点はないか？
  - 例：API 仕様にあるフィールドがデータモデルにない／その逆
- **frontend.md** と **backend.md** の対応は自然か？
  - 画面から呼び出す API が足りない／余計なものがないか
- **architecture.md / infra.md** とアプリの設計が噛み合っているか？

### 2. 抜け漏れ・足りない前提

- MVP を実装する上で、「これは最低限決めておいた方が良い」という前提やルールが抜けていれば指摘してください。
  - 例：エラーハンドリング方針、タイムゾーン、金額・通貨の扱い方 など

### 3. リスク・設計上の懸念

- 今の設計のまま進めたときに、詰まりそうなポイントや将来的に変更が大変になりそうな箇所があれば教えてください。
  - 例：Trips / Expenses のモデリング、共有トリップの扱い、rate_to_jpy の持ち方 など

### 4. 改善提案（あくまで「MVP の範囲」で）

- 「ここはこうしておくとシンプルになる」「ここは将来拡張しやすくなる」という提案があれば、
  - 仕様を大きく変えない範囲で、具体的に提案してください。
- ただし、MVP のスコープを広げすぎる提案（複雑な認証、課金、通知など）は優先度低めの「将来のメモ」として扱ってください。
