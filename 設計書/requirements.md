# 要件定義（MVP）

Travel Expense App の MVP 版における要件をまとめる。

本ドキュメントでは特に「非機能要件」を明確にする。

---

## 1. 機能要件（概要）

詳細は以下のドキュメントを参照：

- `product.md`
- `data-model.md`
- `api.md`
- `frontend-spec.md`

ここでは概要のみ記載する。

### 1.1 機能一覧（MVP）

- トリップ管理
  - 自分のトリップ一覧表示
  - 共有トリップ一覧表示
  - トリップの作成
  - トリップ詳細表示
- 支出管理
  - トリップごとの支出一覧表示
  - 支出の登録
  - 支出の削除
  - 現地通貨での合計表示
  - 円換算（rate_to_jpy を用いた合計表示）
- PWA 対応（ホーム画面追加、簡易オフライン対応）

## 2. 非機能要件

### 2.1 パフォーマンス要件

#### フロントエンド

- 主要画面（トリップ一覧・トリップ詳細）の初回表示は、一般的な 4G 回線＋スマホ環境で **3〜4 秒以内**を目標とする。
- 2 回目以降のアクセスは、PWA のキャッシュにより **体感 2 秒以内**で画面が見えることを目指す。
- 重い画像や大容量アセットは使用しない（アイコンや軽量な画像のみ）。

#### バックエンド

- 通常時の API レスポンス（Lambda 実行）は、**500ms 以内**を目標とする。
- コールドスタートの影響で 1 秒超える場合は許容範囲とする（個人利用のため）。

#### データ量想定

- ユーザー数：数名〜十数名程度（自分と友人）
- トリップ数：ユーザーあたり数十件
- 支出数：トリップあたり数百件までを想定

この規模であれば、DynamoDB + Lambda 構成で十分な性能が出る前提。

---

### 2.2 セキュリティ方針

#### MVP（認証なし）

- MVP 版では一時的に認証を省略し、「限られた友人だけが使うクローズドな環境」を前提とする。
- それでも以下は必須とする：
  - すべての通信は HTTPS 経由（CloudFront / API Gateway）で行う。
  - CORS 設定は、特定のフロントエンドドメインからのアクセスに限定する。
  - デバッグ用ヘッダ（`X-Debug-User-Id` 等）は **本番リリース前に削除する方針**とする。

#### 将来（Cognito 導入後）

- Cognito User Pool によるユーザー管理を行う。
- フロントエンドから API を呼び出す際は、Cognito の ID トークン（JWT）を `Authorization: Bearer <token>` で送信する。
- API Gateway 側で JWT を検証し、Lambda には「検証済みの `user_id`（＝`sub`）」のみを渡す。
- RBAC（Role Based Access Control）
  - 自分のトリップ / 支出のみ閲覧・変更できる
  - 共有トリップは TripMembers テーブルに基づいてアクセス制御

---

### 2.3 オフライン方針（PWA ロードマップ）

#### MVP

- **静的ファイルのみキャッシュ**する。
  - HTML / JS / CSS / アイコン / manifest など。
- オフライン時の挙動：
  - 画面は開けるが、API コールは失敗しエラーメッセージを表示する。
  - 「オフラインです。再接続後にもう一度お試しください」といった文言を表示。
- 支出データやトリップデータをローカルに保持しての編集・同期は **MVP では行わない**。

#### 将来

- オフライン時の入力キューを実装する。
  - ローカルストレージや IndexedDB に「未同期の支出」を保持。
  - オンライン復帰時にまとめて API に送信。
- 簡易なコンフリクト解消：
  - 「サーバー側が新しい場合はサーバー優先」などのポリシーを設計。
- 将来的に「トリップ単位でオフラインキャッシュ」を検討する。

---

### 2.4 コスト管理

- できる限り **無料〜低コスト（数百円/月）** を目標とする。
- サービス選定ポリシー：
  - 常時起動サーバー（EC2 等）は使わず、**Lambda + API Gateway + DynamoDB** のサーバレス構成を基本とする。
  - ストレージは S3 を利用（画像等も最小限）。
  - CloudFront を利用してフロント配信（キャッシュにより API 負荷も軽減）。

#### 開発・検証環境

- `dev` 環境は無料枠内を目安に利用する。
- 不要なリソースは Terraform でまとめて削除できる状態を維持する。

#### 本番環境

- 利用ユーザー・トラフィックともに小規模のため、月数百円〜千円程度を許容範囲とする。
- 定期的に AWS Billing / Cost Explorer でコストを確認する運用とする。

---

### 2.5 運用・監視

- ログ：
  - Lambda のログは CloudWatch Logs に出力する。
  - 重要なエラー（例外）はスタックトレース付きでログ出力する。
- 障害検知（MVP）：
  - 自動アラート（SNS 等）は必須とはしない。
  - 開発者自身が定期的に CloudWatch Logs を確認し、致命的なエラーがないかチェックする。
- データ保護：
  - DynamoDB のバックアップ／復旧手段（オンデマンドバックアップ or PITR）を将来的に検討する。
  - MVP では手動エクスポートでも可。

---

### 2.6 対応ブラウザ・端末

- 対象：
  - iPhone（Safari 最新版）を最優先ターゲットとする。
- 推奨：
  - PC ブラウザ（Chrome / Edge）でも動くことが望ましいが、レイアウト最適化はスマホを優先する。
- OS バージョン：
  - iOS のサポート範囲は「直近数年のバージョン」を目安とする（厳密な OS バージョンのサポート表は持たない）。

---

### 2.7 可用性・スケーラビリティ

- 可用性：
  - 個人利用＋友人利用のため、**24x365 の高可用性は求めない**。
  - 短時間のダウンタイムやデプロイ時の数分停止は許容とする。
- スケーラビリティ：
  - Lambda + DynamoDB のマネージドサービスにより、将来的にユーザー数が増えても基本的にはスケール可能。
  - 大規模負荷を想定したチューニングは不要。

---

## 3. まとめ

- 機能要件は別ドキュメント（product/api/frontend-spec）で詳細定義済み。
- 本ドキュメントでは、特に以下の非機能要件を明確化した：
  - パフォーマンス
  - セキュリティ
  - オフライン方針
  - コスト管理
  - 運用・監視
  - 対応ブラウザ／可用性

これにより、MVP の範囲と「やりすぎないライン」がはっきりし、個人開発として現実的な運用が可能となる。
