# アプリ開発手順（実務エンジニア風）

この文書は、Travel Expense App を実務エンジニアのように開発するための標準的なプロセスを示す。

---

# ① 目的・ゴール定義（What / Why）

- 何のためのアプリか（例：海外旅行中の支出管理）
- 誰のためか（自分・友達）
- MVP で実装すること、しないこと
- product.md にまとめる

---

# ② 要件定義（機能要件＋非機能要件）

## 機能要件（例）

- トリップの作成・一覧・詳細
- 支出の登録・表示・削除
- 円換算（rate_to_jpy）
- 共有トリップの扱い

## 非機能要件（例）

- パフォーマンス：主要画面は 2〜3 秒以内に表示
- セキュリティ：本番では Cognito 認証必須
- オフライン方針：MVP はオンライン前提
- コスト：月額数百円以内を目標
- 運用性：ログ確認方法を決めておく

→ docs/requirements.md を作成すると良い。

---

# ③ 全体設計（アーキテクチャ）

- PWA + AWS サーバレス構成
- フロント：React + TypeScript + Vite
- バックエンド：FastAPI（Lambda）
- インフラ：Terraform
- 使用サービス：
  - API Gateway
  - Lambda
  - DynamoDB
  - S3（フロント配信）
  - CloudFront
  - Cognito（将来）

→ architecture.md に 1 ページでまとめる。

# ⑥ 実装（小さく縦に切って進める）

実務では “一本のユーザーストーリー” を完成させていく。

例：

1. backend：`GET /me/trips`
2. frontend：ホーム画面を作り API をつなぐ
3. backend：`GET /trips/{id}` + `GET /expenses`
4. frontend：トリップ詳細ページ
5. backend：`POST /expenses`
6. frontend：支出追加モーダル

このように「backend → frontend」の縦切りで進めると効率が良い。

# ⑦ テスト（計画 → 実行）

### テスト観点（例）

- トリップが 0 件の場合の表示
- 支出が正しく登録されるか
- 金額のバリデーション
- API エラー時のメッセージ
- スマホ（iPhone）で表示崩れがないか

docs/test-plan.md に簡易メモとして残すと良い。
