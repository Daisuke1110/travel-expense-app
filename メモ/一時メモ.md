以下の6ステップに分けるのが妥当です。
まずは Phase 1 = プロフィール基盤、次に Phase 2 = 名前表示、最後に Phase 3 = 招待基盤 です。

Step 1. バックエンドで GET /me を作る 〇
目的:

- ログイン済みユーザーの sub, email, name を返す
- Users にレコードが無ければ自動作成する

やること:

1. auth.py で JWT claims を取る関数を追加
2. routes/me.py を新規作成
3. GET /me 実装
4. main.py で router 登録

完了条件:

- GET /me で user_id, email, name が返る
- 初回アクセス時に Users が自動作成される

———

Step 2. バックエンドで PATCH /me を作る 〇
目的:

- 初回ログイン後に表示名を登録できるようにする

やること:

1. MeUpdateRequest を追加
2. PATCH /me 実装
3. name のバリデーション追加

完了条件:

- PATCH /me で name を保存できる
- 再度 GET /me で更新値が返る

———

Step 3. フロントでプロフィール設定画面を作る  
 目的:

- name 未設定のユーザーが最初にプロフィールを設定できるようにする

やること:

1. frontend/src/api/me.ts を追加
2. frontend/src/pages/ProfileSetupPage.tsx を追加
3. 名前入力フォームと保存処理を実装

完了条件:

- 画面から name を保存できる
- 保存後に通常画面へ戻れる

———

Step 4. App.tsx に初回プロフィール完了導線を入れる  
 目的:

- Hosted UI ログイン後、name 未設定なら必ずプロフィール設定画面に送る

やること:

1. 起動時に fetchMe() を呼ぶ
2. name が空なら /profile/setup
3. そうでなければ通常ルーティング

完了条件:

- 初回ログインユーザーはプロフィール設定画面へ遷移
- name 設定済みユーザーは通常どおり使える

———

Step 5. APIレスポンスに name を含める  
 目的:

- 画面表示を sub から name 優先に変えるためのデータを返す

やること:

1. GET /trips/{tripId}/members に name を追加
2. GET /trips/{tripId}/expenses に paid_by_name を追加
3. 必要なら user_name も追加
4. Users テーブルを batch_get してマージ

完了条件:

- members/expenses API が name 系フィールドを返す

———

Step 6. フロント表示を name 優先に切り替える  
 目的:

- 画面上で sub を見せないようにする

やること:

1. Trip メンバー一覧を name ?? user_id
2. Expense の Paid by を paid_by_name ?? paid_by_user_id
3. Paid totals も name 優先
4. 必要なら owner 表示も調整

完了条件:

- 通常画面では sub ではなく name が見える
- name 未設定時のみ fallback で user_id を表示

———

その後の将来ステップ  
 Step 7. email 招待基盤  
 目的:

- メンバー追加を user_id 手入力から email ベースに移行する

これは Step 1-6 完了後にやるのが安全です。

———

おすすめの開始順  
 今すぐ着手するなら、

1. Step 1
2. Step 2
3. Step 3
4. Step 4  
   の順です。

つまり最初は 「プロフィールを持てるようにする」 ところまでで一区切りです。  
 名前表示切替はその次です。

必要なら次に Step 1 だけ を、編集するファイルごとに具体コードレベルで分解します。
